# sayo-backend-system

> Англоязычная версия: [README.md](README.md)

Распределённый бэкенд **распознавания речи (STT)** для Sayo. Предоставляет клиентам API по **gRPC**, направляет потоковые сессии через кластер **Ray** и выполняет каждую акустическую модель в контейнере **Docker**, который подключается к Ray как **тяжёлый актёр** (одна загруженная модель на контейнер, при необходимости — несколько параллельных сессий на одну модель).

Цели проектирования:

- **Тонкий узел управления**: шлюз, маршрутизатор, менеджер узлов и реестр моделей работают в небольших образах без PyTorch и без весов конкретных моделей.
- **Самодостаточные исполнители**: после регистрации узлы обработки получают один **образ актёра** на модель; на горячем пути не требуется общий NFS или объектное хранилище для весов.
- **Чёткое разделение пакетов**: `sayo_host` (оркестрация) не должен импортировать `sayo_image` (код стороны актёра); при расширении системы эту границу нужно сохранять.

Развёрнутое архитектурное описание на русском: [`../server.md`](../server.md).

## Архитектура (обзор)

| Компонент | Назначение |
|-----------|------------|
| **Шлюз (Gateway)** (`sayo_host/gateway`) | Точка входа gRPC: `HealthCheck`, `StreamingRecognize`. Общается с реестром по HTTP и с Ray — для маршрутизации и вызовов актёров. |
| **Главный маршрутизатор (MasterRouter)** (`sayo_host/router`) | Отсоединённый именованный актёр Ray: размещение сессий, повторное использование актёров, учёт видеопамяти, вытеснение простаивающих по LRU. Кэширует манифесты моделей из реестра. |
| **Менеджер узла (WorkerManager)** (`sayo_host/worker_manager`) | Один актёр Ray на физический узел: запуск и остановка контейнеров актёров через Docker; периодическая отчётность о GPU и списке актёров маршрутизатору. |
| **Реестр моделей** (`sayo_host/registry`) | Сервис FastAPI: каталог манифестов, административная регистрация с **обёрткой** образа «одна модель» в **образ актёра**, при необходимости выгрузка во внутренний реестр образов. |
| **Актёр распознавания (TranscriptActor)** (`sayo_image/…`) | Выполняется **внутри** образа актёра: загрузка модели, по желанию детектор речи Silero (ONNX), поток гипотез обратно в шлюз. |

**Путь запроса (упрощённо):** клиент → шлюз → `MasterRouter.acquire_session` → (при необходимости) `WorkerManager.spawn` → контейнер поднимает процесс Ray и `TranscriptActor` → шлюз: `open_session` / `feed` / `next_result` / `close_session` → `release_session` у маршрутизатора.

## Структура репозитория

```
sayo-backend-system/
  proto/                      # контракт gRPC (клиент ↔ шлюз)
    sayo.proto                # источник правды (правки → пересборка заглушек)
    sayo_pb2.py               # сгенерировано — не править вручную
    sayo_pb2_grpc.py          # сгенерировано — не править вручную
  sayo_host/                  # сервисы узла управления — не импортировать sayo_image
    common/                   # журналирование, общие помощники по окружению
    gateway/                  # сервер gRPC
    router/                   # загрузка MasterRouter + сам актёр
    worker_manager/           # загрузка WorkerManager + сам актёр
    registry/                 # приложение FastAPI, каталог, конвейер обёртки, админ-интерфейс
  sayo_image/                 # только в образе актёра
    model_repository/         # model.yaml → адаптер (контракт BaseSTTModel)
    transcript_actor/         # актёр Ray, вспомогательная обработка аудио, точка входа bootstrap
    vad/                      # обёртка Silero ONNX
  vad-weights/                # silero_vad.onnx (копируется в образ актёра)
  docker/                     # Dockerfile для всех запускаемых образов
  deploy/
    single-node/              # docker-compose.yml (топология разработки / демо по умолчанию)
    distributed/              # заметки по многоузловому режиму + черновик cluster.yaml
  tests/smoke/                # сквозной smoke-скрипт
  sayoctl                     # CLI к административному HTTP API реестра
  pyproject.toml
  requirements.txt            # экспорт с фиксацией версий (uv); опционально для установки через pip
```

## Требования

- **Docker** и **Docker Compose** v2 для рекомендуемого сценария.
- **Python ≥ 3.12**, если вы запускаете `sayoctl`, smoke-тесты или локальные инструменты вне контейнеров.
- **NVIDIA Container Toolkit**, если используются модели на GPU и задана переменная `SAYO_NUM_GPUS` (см. compose).
- Образы «одна модель» должны быть собраны так, чтобы реестр мог прочитать **`MODEL_NAME`** из переменных окружения образа и найти `model.yaml` по пути `/app/models/<MODEL_NAME>/` внутри образа (см. конвейер обёртки в `sayo_host/registry/wrapper.py`).

**Согласование версий Python и Ray:** узел-координатор Ray и любой контейнер с актёром должны использовать **совместимые сборки Ray и Python**. Несовпадение версий между координатором и присоединяющимся воркером часто приводит к ошибкам подключения или сериализации.

## Быстрый старт (один узел)

Из каталога `sayo-backend-system`:

```bash
# Сборка и запуск: координатор Ray, реестр, маршрутизатор, менеджер узла, шлюз
docker compose -f deploy/single-node/docker-compose.yml up -d --build
```

**Проброшенные порты** по умолчанию (хост → контейнер):

| Порт | Сервис |
|------|--------|
| 50051 | шлюз gRPC |
| 8000 | реестр моделей (HTTP) |
| 8081 | админ HTTP маршрутизатора (снимок кластера) |
| 8082 | админ HTTP менеджера узла |
| 6379 | Ray GCS / внутренний порт клиента |
| 10001 | Ray Client (`ray://…`) |
| 8265 | панель наблюдения Ray |

Зарегистрируйте образ **«одна модель»**; реестр собирает/помечает **образ актёра** и записывает манифест:

```bash
./sayoctl register-model registry.example.com/sayo-model-nemo:1.0.0
# или локальный тег, если образ уже есть у демона Docker:
./sayoctl register-model my-nemo:1.0.0 --local
```

Smoke-тест (ожидается хотя бы одна зарегистрированная модель):

```bash
python tests/smoke/run.py --host localhost --port 50051 --duration 4
```

Подробнее: [`tests/smoke/README.md`](tests/smoke/README.md).

### Сборка `actor-runtime` (желательно не пропускать)

Регистрация актёра вызывает `docker build` по `docker/Dockerfile.actor`, который ожидает уже собранный образ **`sayo-actor-runtime:latest`** на том же демоне Docker (колёса зависимостей и сгенерированные заглушки proto). При изменении зависимостей или `proto/sayo.proto` пересоберите:

```bash
docker compose -f deploy/single-node/docker-compose.yml --profile build-only build actor-runtime
```

## Клиентский API (gRPC)

Описан в **`proto/sayo.proto`** (после правок перегенерируйте заглушки).

- **`HealthCheck`** — пустой запрос; в ответе поля `ready`, `message` и повторяющийся список `ModelDescriptor`, собранный из манифестов реестра (на стороне шлюза — `GET /v1/models`).
- **`StreamingRecognize`** — **двусторонний поток**:
  - первое сообщение клиента **обязано** быть `StreamingConfig` (идентификатор модели, язык, признак промежуточных результатов, формат аудио, настройки детектора речи);
  - далее — сырые порции **моно PCM** (`bytes`) в соответствии с конфигурацией;
  - сервер отдаёт поток `StreamingRecognizeResponse` (`transcript`, `is_final`, `confidence`, словарь строк `metadata`). Ранние ответы могут содержать подсказки о ходе установления соединения в `metadata["connection_status"]` (см. комментарии в proto-файле).

Пересборка Python-заглушек после изменения proto (зависимость для разработки `grpcio-tools`, та же мажорная версия, что у `grpcio`):

```bash
cd sayo-backend-system
uv sync --group dev
uv run python -m grpc_tools.protoc -I . --python_out=. --grpc_python_out=. proto/sayo.proto
```

Файлы `proto/sayo_pb2.py` и `proto/sayo_pb2_grpc.py` рядом с `sayo.proto` перезаписываются; это согласовано с импортом `from proto import sayo_pb2` в коде.

## Реестр моделей (HTTP)

Публичное API чтения:

- `GET /v1/health` — работоспособность и число моделей.
- `GET /v1/models` — список манифестов (JSON).
- `GET /v1/models/{model_id}` — один манифест.

Административное API:

- `POST /v1/admin/models` — регистрация (`image_ref`, опционально `force`, `local_only`).
- `POST /v1/admin/models/register-stream` — то же с потоком прогресса в формате **NDJSON**.
- `DELETE /v1/admin/models/{model_id}` — удаление из каталога (опциональное удаление образа — через параметры запроса, где реализовано).
- `GET /v1/admin/state` — отладочный снимок состояния.
- `GET /admin` — небольшой HTML-интерфейс (если включён в поставку).

Поля **манифеста** (концептуально): `model_id`, `actor_image_tag`, `model_dir`, параметры аудио и среды исполнения, `min_vram_gb`, `max_concurrent_sessions`, блок `vad` и т.д. Шлюз и маршрутизатор получают манифесты **только по HTTP**, без чтения YAML с диска на своей стороне.

**Распределённый режим:** задайте для сервиса реестра `INTERNAL_DOCKER_REGISTRY`, чтобы собранные образы актёров **отправлялись** в ваш частный реестр; вычислительные узлы выполняют **загрузку** при первом `spawn`. См. [`deploy/distributed/README.md`](deploy/distributed/README.md).

## `sayoctl` (интерфейс командной строки)

Устанавливается как консольная команда из `pyproject.toml` (`pip install -e .` или `uv sync`). Базовый URL реестра задаётся переменной `SAYO_REGISTRY_URL` (по умолчанию `http://localhost:8000`).

Типичные команды:

```bash
sayoctl register-model <image_ref> [--local] [--force]
sayoctl list
sayoctl unregister <model_id> [--remove-image]
sayoctl state
```

## Переменные окружения (справочник)

Ниже — типичные **значения по умолчанию или из compose**; при необходимости переопределяйте в своём окружении.

| Переменная | Где используется | Смысл |
|------------|------------------|--------|
| `RAY_ADDRESS` | шлюз, маршрутизатор, менеджер узла | Адрес клиента Ray, напр. `ray://ray-head:10001`. |
| `RAY_NAMESPACE` | все клиенты Ray | Логическое пространство имён для именованных актёров (по умолчанию `sayo`). |
| `REGISTRY_URL` | шлюз, маршрутизатор, менеджер узла | Базовый URL HTTP API реестра моделей. |
| `NODE_ID` | менеджер узла | Уникальный идентификатор узла (`WorkerManager:<NODE_ID>`). |
| `RAY_HEAD_HOST` | менеджер узла, командная строка актёра | Адрес Ray, передаваемый в контейнеры актёров, напр. `ray-head:6379`. |
| `SAYO_NETWORK` | менеджер узла | Имя сети Docker для запускаемых контейнеров актёров (в compose может быть `sayo_sayo_net`). |
| `SAYO_DISTRIBUTED` | менеджер узла | Значение `1` — выполнять `docker pull` образа актёра перед запуском. |
| `GATEWAY_PORT` | шлюз | Порт прослушивания gRPC (по умолчанию `50051`). |
| `REGISTRY_ROOT` | реестр | Каталог каталога манифестов (по умолчанию `/manifests`). |
| `INTERNAL_DOCKER_REGISTRY` | реестр | Если задано — выгрузка собранных образов актёров в этот реестр. |
| `SAYO_REPO_ROOT` | сборка в реестре | Корень файловой системы с каталогами `sayo_image`, `docker`, `proto`. |
| `SAYO_ACTOR_RUNTIME_IMAGE` | реестр | Переопределение тега образа actor-runtime при сборке обёртки. |
| `SAYO_ACTOR_REGISTER_TIMEOUT` | менеджер узла | Секунды ожидания регистрации актёра в Ray после `docker run`. |
| `SAYO_ACTOR_SHM_SIZE` | менеджер узла | Параметр Docker `shm_size` для контейнеров актёров (по умолчанию `2g`). |
| `SAYO_DISABLE_VAD` | шлюз (и актёр) | Истинное значение — игнорировать VAD клиента / отключить VAD для отладки. |
| `SAYO_REPLACE_MASTER_ROUTER` | загрузка маршрутизатора | Истинное значение — завершить существующий `MasterRouter` перед пересозданием (опасно в эксплуатации). |
| `ADMIN_PORT` | маршрутизатор, менеджер узла | HTTP-порт опционального сервера админ-снимка. |

## Несколько узлов (распределённый режим)

Топология второго этапа: на одном **мастер**-хосте — координатор Ray, шлюз, маршрутизатор, реестр и менеджер узла; на дополнительных **воркер**-хостах — процессы Ray и свой менеджер узла с отдельным `NODE_ID`. Воркерам не обязательны контейнеры шлюза и реестра, если с мастера доступны адрес Ray и реестр образов.

Пошаговые заметки: [`deploy/distributed/README.md`](deploy/distributed/README.md).

## Локальная разработка (без полного compose)

```bash
uv sync
# или: pip install -e .
```

Линтеры и форматирование (см. `pyproject.toml` и `.pre-commit-config.yaml`):

```bash
pre-commit run --all-files
```

При доработках сохраняйте **независимость `sayo_host` от `sayo_image`**, чтобы образы узла управления оставались лёгкими, а образы актёров — единственным местом для адаптеров моделей и тяжёлых зависимостей.

## Устранение неполадок (кратко)

- **В `HealthCheck` у шлюза `ready=false`** — реестр недоступен с точки зрения шлюза; проверьте `REGISTRY_URL` и сеть.
- **Сессия «висит» или актёр не появляется** — панель Ray (`8265`), админ маршрутизатора (`8081`), админ менеджера узла (`8082`); убедитесь, что в манифестах указан `actor_image_tag` и демон на воркере может загрузить и запустить этот образ.
- **Актёр не регистрируется в Ray** — часто несовпадение образа / Python / Ray, слишком малый `shm_size` или несогласованность планирования GPU (`device` / `num-gpus`) с узлом.
- **Падает сборка при регистрации** — убедитесь, что образ `sayo-actor-runtime:latest` существует (профиль compose `build-only`) и к реестру смонтирован сокет Docker.
